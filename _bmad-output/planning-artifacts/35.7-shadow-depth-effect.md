# Story 35.7 Planning: Shadow/Depth Effect for 3D Door Appearance

## Adopted Approach: Post-Render Shadow Wrapper

### Strategy
Add shadow as a **post-processing step** applied after each theme's `Render()` output. A shared helper function `applyShadow()` appends a shadow column (`▐`) to the right of each rendered line and a shadow row (`▄`) below the threshold/floor line.

### Rationale
- **DRY**: Single shadow implementation shared across all 4 themes, avoiding duplicating shadow logic in each theme's render function
- **Consistent**: Guarantees identical shadow behavior across Classic, Modern, Sci-Fi, and Shoji
- **Non-invasive**: Themes render exactly as before; shadow is layered on top
- **Graceful degradation**: Width check happens once in the wrapper, not per-theme

### Implementation Plan
1. Create `shadow.go` with `ApplyShadow(rendered string, width int, minWidth int, selected bool) string`
2. Call `ApplyShadow` from each theme's render closure after generating the door output
3. Shadow characters:
   - Right column: `▐` (unselected), `█` (selected — full block for enhanced depth)
   - Bottom row: `▄` repeated across `width + 1` (door width + shadow column)
   - Shadow offset: bottom row starts 1 char to the right (no shadow in top-left corner)
4. Width guard: if `width < minWidth + 2`, return rendered string unchanged
5. Compact mode: no shadow (height < MinHeight already uses card style)
6. Update golden files by running tests with `-update`
7. Add shadow-specific unit tests

### Shadow Color
- Shadow uses a dim/muted color (not the theme's frame color)
- Unselected: Color 240 (medium gray) — subtle depth cue
- Selected: Color 250 (lighter gray) — "lifted" appearance

## Rejected Options

### Option A: Per-Theme Shadow Integration
Embed shadow rendering directly into each theme's row-by-row loop.

**Rejected because:**
- Duplicates shadow logic 4 times (once per theme)
- Each theme would need to modify its width calculations
- Risk of inconsistent shadow behavior across themes
- More code to maintain

### Option B: Shadow via Lipgloss Border Style
Use Lipgloss border/margin styling to create shadow effect.

**Rejected because:**
- Lipgloss borders don't support half-block characters
- Would change the rendering paradigm (themes use manual string building)
- Less control over shadow character selection
- Can't differentiate selected/unselected shadow styles easily

### Option C: Shadow in DoorsView (caller level)
Apply shadow in `doors_view.go` after calling `theme.Render()`.

**Rejected because:**
- Breaks encapsulation — shadow is a visual theme concern, not a layout concern
- Door width passed to theme wouldn't account for shadow overhead
- Would need to adjust lipgloss.JoinHorizontal spacing
- Golden file tests call `theme.Render()` directly and wouldn't capture shadow

### Option D: Full-width shadow row spanning all 3 doors
Single shadow row under all doors instead of per-door shadow.

**Rejected because:**
- Loses per-door 3D effect
- Selected door shadow enhancement wouldn't work
- Doesn't match the AC requirement for per-door shadow rendering

# Story 39.11: Non-Door View Inline Hints

**Epic:** 39 — Keybinding Display System
**Status:** Not Started
**Priority:** P1
**Estimate:** M (3-5 hours)
**Depends On:** 39.9

---

## User Story

As a new user navigating beyond the doors view, I want to see inline key labels on interactive elements in detail view, search, mood picker, and other views, so that I always know what I can do without memorizing key bindings.

## Context

Story 39.10 handles the flagship doors view. This story extends inline hints to all other views where the user interacts with actionable elements. Each view has different interaction patterns and needs view-appropriate hint placement.

Design reference: [Default Tooltips Party Mode](../../_bmad-output/planning-artifacts/default-tooltips-mode-party-mode.md)

## Acceptance Criteria

### AC-1: Detail View Hints
- [ ] `[esc] Back` hint near the top or navigation area
- [ ] `[c] Complete` hint near the task status/action area
- [ ] `[b] Block` hint near the task status/action area
- [ ] `[e] Expand` and `[f] Fork` hints near relevant action areas
- [ ] Hints sourced from keybinding registry (39.1) for the DetailView mode
- [ ] Hints conditioned on `show_inline_hints` config

### AC-2: Search View Hints
- [ ] `[enter] Search` and `[esc] Cancel` hints near the search input field
- [ ] `[↑/↓] Navigate` hints near the results list when results are visible
- [ ] Hints appropriate for the text input context

### AC-3: Mood View Hints
- [ ] Numbered labels (`1`, `2`, `3`, etc.) rendered next to each mood emoji option
- [ ] Hints make the numbered selection mechanism discoverable

### AC-4: Add Task View Hints
- [ ] `[enter] Submit` and `[esc] Cancel` hints near the text input
- [ ] Consistent with Search View hint placement pattern

### AC-5: Other Views
- [ ] Health/Stats view: relevant navigation hints
- [ ] Help view: minimal hints (the view itself IS the help)
- [ ] Command mode: no inline hints (bar/overlay sufficient)
- [ ] Each view's hints match its keybinding registry entry

### AC-6: Consistent Styling
- [ ] All inline hints use the same `renderInlineHint()` function from 39.9
- [ ] Hint brightness/dimming consistent across all views
- [ ] Fade mode (session N-1) applies uniformly

### AC-7: Tests
- [ ] Each view renders correct hints when enabled
- [ ] Each view renders no hints when disabled
- [ ] Hints sourced from keybinding registry (no hardcoded keys)
- [ ] Golden file tests for Detail view and Mood view with hints
- [ ] Table-driven tests following project testing standards

### AC-8: Race Detector
- [ ] `go test -race ./internal/tui/...` passes (MANDATORY per CLAUDE.md)

## Technical Notes

- Changes span multiple view files: `detail_view.go`, `search_view.go`, `mood_view.go`, `add_task_view.go`, etc.
- Each view's View() method needs access to the `show_inline_hints` config state
- The hint placement is view-specific — no single pattern fits all views
- Consider passing `showInlineHints bool` to view SetHeight/SetWidth or via a shared config struct
- Can be worked in parallel with 39.10 since both depend on 39.9 infrastructure

## Dependencies

- Story 39.9 (Inline Hint Rendering Infrastructure) — provides renderInlineHint()
- Story 39.1 (Keybinding Registry) — data source for per-view hints
